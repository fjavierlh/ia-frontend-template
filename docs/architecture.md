# Multi-agent development system architecture

## Overview

AI-assisted development system composed of 4 specialized agents orchestrated via a coding agent CLI (Claude Code, OpenCode, or compatible). A human (Tech Lead) supervises the process and acts as a validation gate at key points.

Each project template (frontend or backend) includes its own independent agent system instance, with instructions adapted to the specific stack and structure.

## Tech stack

- **Runtime**: Bun 1.x
- **Language**: TypeScript 6.0 (strict mode)
- **Framework**: React 19 + Vite 8 (frontend)
- **Architecture**: Hexagonal (ports & adapters) with DDD and vertical slicing by business module
- **Agent engine**: Agnostic — compatible with Claude Code (`claude -p`) and OpenCode (`opencode -p`). Instruction files use the `CLAUDE.md` standard
- **Work management**: Linear (official MCP server, HTTP transport)
- **Monitoring**: Sentry (official MCP server, HTTP transport, read-only)
- **Version control**: GitHub with `gh` CLI (no MCP, more token-efficient). Feature branching + atomic conventional commits
- **Testing**: BDD with Playwright + Gherkin, TDD with Vitest
- **Static analysis**:
  - `tsc` — TypeScript compilation as the first line of validation
  - Biome — Unified formatting, linting, and import sorting
  - Knip — Dead code detection, unused exports, unnecessary dependencies
  - SonarCloud — Code smells, vulnerabilities, tech debt (CI)

## System roles

### Tech Lead (human)

Supervisory role. Does not execute tasks directly but:

- Approves or rejects user stories generated by the CEO
- Reviews the Planner's plan (including Gherkin scenarios) before the Coder starts
- Can intervene at any time to redirect work
- Has the final word on merges to the main branch
- Writes user stories directly when needed
- **Authorizes any modification to existing tests** — no agent may alter a previously-passing test without explicit Tech Lead approval

### CEO agent

Responsible for product vision. Operates in two modes:

- **Reactive mode**: The human provides a problem, need, or idea and the CEO transforms it into a structured user story with acceptance criteria
- **Proactive mode**: Analyzes the current product state by reading `.feature` files (Gherkin) to understand what functionality exists, queries Sentry to detect errors, and reviews issues in Linear to identify gaps

**Efficiency note**: The CEO **does not read source code**. It understands the product through Gherkin scenarios, which act as executable living documentation. This saves tokens significantly and gives a product-level (not technical) view of the system state.

**Output**: User story in standard format, created as a Linear issue with:
- Descriptive title
- Description with context and motivation
- Acceptance criteria (Given/When/Then)
- Suggested priority
- Relevant labels

### Planner agent

Receives an approved user story and decomposes it into an executable plan:

- Analyzes existing code to understand context
- Identifies affected files and dependencies
- **Writes Gherkin scenarios** (`.feature` files) as the first planning action — this eliminates the gap between requirements and tests, and allows the CEO and Tech Lead to validate scenarios before implementation
- Creates sub-issues in Linear with ordered tasks
- Defines execution order and inter-task dependencies

**Output**: Gherkin scenarios + ordered task list in Linear, each with:
- Clear description of what to do
- Files to create or modify
- Completion criteria
- Dependencies with other tasks

### Coder agent (driver + navigator)

Executes plan tasks following a strict XP and TDD cycle (see xp-tdd-practices skill):

1. **REASON**: Reads the task, analyzes the code, creates a case list as TODOs in the test file
2. **RED**: Writes the minimal failing test (simplest case first)
3. **GREEN**: Implements the minimal code following TPP (Transformation Priority Premise)
4. **REFACTOR**: Cleans up code with tests green
5. **COMMIT**: Atomic commit with conventional commits (always green post-refactor)
6. **RE-EVALUATE**: Reviews the list, reorders if necessary, next case

Development is always **inside-out**: Domain → Use Case → Infrastructure

**Coder constraints**:
- Never writes production code without a test justifying it
- Respects hexagonal architecture: pure domain, ports as interfaces, adapters in infrastructure
- Uses monads (Either, Maybe) for expressive domain error handling
- Atomic and descriptive commits (conventional commits) — one commit per green+refactor cycle
- Updates task status in Linear at start and completion
- **Never modifies an existing test that was passing** without explicit Tech Lead authorization. If a new implementation breaks an existing test, stops and consults

### Reviewer agent

Reviews each completed task against multiple criteria:

- **Requirements**: Are all acceptance criteria met? Do Gherkin scenarios pass?
- **Tests**: Adequate coverage? Meaningful tests (sociable, not solitary with excessive mocks)? InMemory repositories used instead of mocks?
- **Architecture**: Is hexagonal separation respected? Is domain pure? Validated with ArchUnitTS?
- **Code**: Consistent naming? SOLID principles? Monads used correctly?
- **Static analysis**: Runs `tsc`, Biome, Knip and reviews SonarCloud metrics
- **Runtime**: Queries Sentry to verify no new errors

**Output**: One of two results:
- **Approved**: Marks the task as completed in Linear, ready for merge
- **Rejected**: Returns to Coder with specific comments (what fails, where, correction suggestion)

## Integrations

### Linear MCP server

```bash
# Claude Code
claude mcp add --transport http linear-server https://mcp.linear.app/mcp

# OpenCode
opencode mcp add linear-server --url https://mcp.linear.app/mcp
```

Used by: CEO (create issues), Planner (create sub-tasks), Coder (update status), Reviewer (approve/reject)

### Sentry MCP server

```bash
# Claude Code
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# OpenCode
opencode mcp add sentry --url https://mcp.sentry.dev/mcp
```

Used by: CEO (identify production issues), Reviewer (verify no new errors)

Note: Sentry MCP is read-only. It cannot create or modify data in Sentry.

### Playwright MCP server

```bash
# Claude Code
claude mcp add playwright --scope project -- npx -y @playwright/mcp --headless

# OpenCode
opencode mcp add playwright -- npx -y @playwright/mcp --headless
```

Used by: Coder (run e2e tests during implementation), Reviewer (verify Gherkin scenarios pass)

Note: Requires browsers installed (`npx playwright install chromium`). Runs in headless mode for agent use.

### Context7 MCP server

```bash
# Claude Code
claude mcp add context7 -- npx -y @upstash/context7-mcp@latest

# OpenCode
opencode mcp add context7 -- npx -y @upstash/context7-mcp@latest
```

Used by: Planner (check docs when analyzing code), Coder (up-to-date library documentation during implementation), Reviewer (verify current APIs are used)

Note: No API key required for basic usage. Provides up-to-date, version-specific documentation for any public library. For higher rate limits, a free key can be obtained at context7.com.

### GitHub (`gh` CLI)

MCP is not used for GitHub. The `gh` CLI is more token-efficient and offers full control over the Git workflow.

```bash
# Coder creates feature branches
gh issue develop <issue-number> --checkout

# Reviewer can inspect PRs
gh pr view <pr-number>

# Merge after Tech Lead approval
gh pr merge <pr-number> --squash
```

**Branching strategy**: Feature branching from `main`. Each task has its own branch (`feat/LINEAR-123-description`). Merge to `main` only after Reviewer + Tech Lead approval.

## System file structure

```
project-root/
├── CLAUDE.md                  # General instructions (compatible OpenCode + Claude Code)
├── .claude/
│   ├── agents/                # Subagent definitions (auto-delegated, worktree-isolated)
│   │   ├── ceo.md
│   │   ├── planner.md
│   │   ├── coder.md
│   │   └── reviewer.md
│   └── skills/                # Reusable knowledge shared across agents
│       ├── xp-tdd-practices/SKILL.md
│       ├── hexagonal-patterns/SKILL.md
│       ├── gherkin-writing/SKILL.md
│       ├── conventional-commits/SKILL.md
│       └── user-story-format/SKILL.md
├── docs/
│   └── architecture.md        # This document
├── features/                  # Gherkin scenarios (BDD)
│   ├── auth/
│   │   └── login.feature
│   └── health/
│       └── check.feature
└── src/                       # Source code (see AGENTS.md for structure)
```

Each agent runs in its own **isolated git worktree** (`isolation: worktree` in frontmatter) with a **forked context** (`context: fork`), ensuring a clean context window per task and no state leakage between agents. Claude Code automatically delegates to the appropriate agent based on the task description and context.

## Workflow

### 1. Work entry

Two possible paths:

**Path A — CEO generates story**:
The CEO reads existing `.feature` files and queries Sentry to propose improvements:
```bash
# Claude Code
claude -p "Analyze existing features and Sentry, propose improvements" \
  --max-turns 10 \
  --allowedTools "mcp__linear-server__*,mcp__sentry__*,Read,Grep"

# OpenCode
opencode -p "Analyze existing features and Sentry, propose improvements"
```

**Path B — Human creates story**:
The Tech Lead creates the issue directly in Linear or asks the CEO to formalize it.

### 2. Tech Lead validation

The human reviews the story in Linear. If approved, moves it to "Ready for Planning".

### 3. Planning (includes Gherkin)

The Planner decomposes the story and writes the Gherkin scenarios:
```bash
claude -p "Plan story [LINEAR-ID]: decompose into tasks and write .feature files" \
  --max-turns 15 \
  --allowedTools "mcp__linear-server__*,Read,Write,Grep,Glob"
```

The Tech Lead reviews both the plan and the Gherkin scenarios before giving the green light.

### 4. Implementation (TDD inside-out)

```bash
claude -p "Implement task [LINEAR-TASK-ID] following the TDD cycle" \
  --max-turns 30 \
  --allowedTools "mcp__linear-server__*,Read,Write,Edit,Bash(bun test:*),Bash(bunx knip),Bash(gh *)"
```

Each green+refactor cycle produces an atomic commit.

### 5. Review

```bash
claude -p "Review task [LINEAR-TASK-ID]" \
  --max-turns 15 \
  --allowedTools "mcp__linear-server__*,mcp__sentry__*,Read,Grep,Glob,Bash(bun test:*),Bash(bunx knip),Bash(bun run check)"
```

If the Reviewer rejects, returns to step 4 with the comments.

### 6. Closure

Once approved by the Reviewer, the Tech Lead merges:
```bash
gh pr merge <pr-number> --squash
```

The task is marked as Done in Linear.

## State transition rules

| From | To | Condition | Who decides |
|------|----|-----------|-------------|
| New | Ready for Planning | Story approved | Tech Lead |
| Ready for Planning | In Planning | Planner assigned | Automatic |
| In Planning | Ready for Dev | Plan + Gherkin approved | Tech Lead |
| Ready for Dev | In Progress | Coder assigned | Automatic |
| In Progress | In Review | Tests pass, Coder completes | Coder |
| In Review | In Progress | Reviewer rejects | Reviewer |
| In Review | Done | Reviewer approves + Tech Lead merges | Tech Lead |

## Design principles

1. **The human is always in control**: No agent can merge to main or deploy without Tech Lead approval
2. **Full traceability**: Everything is recorded in Linear (decisions, rejections, plan changes)
3. **Gherkin as common language**: The `.feature` scenarios are the living documentation that all agents and the human understand
4. **Existing tests are sacred**: No agent modifies a passing test without Tech Lead authorization
5. **Fail-safe**: If an agent fails or gets stuck, the Tech Lead can intervene manually
6. **Incrementality**: Each task is small and self-contained. One commit per green+refactor cycle
7. **Quality as a gate, not an afterthought**: The Reviewer is mandatory, not optional
8. **Agent-engine agnostic**: Instructions work with any MCP-compatible coding agent
